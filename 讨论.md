## 4. 已完成的总体认知修正

### 4.1 关于“地面站”的理解修正

一开始容易把地面站理解为：
- 显示地图
- 显示数据
- 串口收 MAVLink
- 能连设备

后来逐步形成更准确理解：

真正的地面站至少包括以下几类能力：

1. **实时监控**
   - 位置、姿态、速度、电池、GPS、链路状态、模式、传感器状态等

2. **控制操作**
   - Arm / Disarm
   - 模式切换
   - Guided / Auto / RTL / Hold 等控制
   - 可能还有手动控制或辅助控制入口

3. **配置**
   - 参数读写
   - 参数分组/向导化
   - 传感器/设备配置流程

4. **任务规划**
   - Waypoint 规划
   - Mission 上传/下载
   - 当前任务状态反馈

5. **健康检查 / 告警**
   - Preflight / pre-mission checklist
   - 电池、GPS、通信、EKF 等异常提示

6. **日志与回放**
   - 记录过程
   - 复盘
   - 调试

### 4.2 对 MavDeck 的核心判断

MavDeck 已经具备一些非常有价值的基础能力，但还不能直接等价于一个成熟的完整地面站。

当前对 MavDeck 的总体判断是：

#### 它已有的能力
- Web 原生方向
- MAVLink 解码/显示基础
- WebSerial / 浏览器侧连接思路
- 地图
- 图表
- 遥测展示
- 参数相关基础能力
- 日志记录 / 回放
- PWA / Worker 等 Web 基础设施

#### 它当前不足的地方
- 缺少完整“操作闭环”
- 缺少成熟的模式切换 / 控制面板
- 缺少任务规划闭环
- 缺少参数流程化配置体验
- 缺少任务前检查 / 健康检查 / 安全提示
- 缺少像成熟 GCS 那样完整的页面与功能组织

### 4.3 已形成的关键表述

> `MavDeck` 已经具备了 Web 地面站的基础设施，但尚未形成完整的“监控 + 操作 + 配置 + 任务规划”的闭环。

## 5. 为什么最后确定基于 MavDeck 改造

这个问题现在已经得到 mentor 认可，方向明确：

> **就基于 `MavDeck`，尽量复现 `QGroundControl` 的全部核心内容。**

### 5.1 选择 MavDeck 的原因

1. **方向最符合课题**
   - Web 原生
   - 和浏览器侧能力结合更自然
   - 比桌面方案更契合“Web 地面站方法改进”

2. **已有较好的技术底座**
   - 不需要从零做协议解码、地图、图表、参数等底层

3. **改造成本相对可控**
   - 相比从零重写，更快进入“功能补全”和“交互升级”

4. **比其他 Web 项目更适合作为主线**
   - `WebGCS` 偏轻量/概念化
   - `electronWebGCS` 偏桌面封装
   - `ground-station` 的主场景和目标不完全一致

### 5.2 当前确定后的表述

当前不再只是“倾向”，而是已经可以明确表述为：

> 第一阶段开始，整体路线就按 **基于 `MavDeck` 做主改造底座** 来推进，并以“尽量复现 `QGroundControl` 的核心能力”为目标。

## 6. 为什么以 QGroundControl 作为主要参考对象

这个结论也已经比较稳定。

### 6.1 当前结论
- `QGroundControl`：作为**主要参考对象**
- `MissionPlanner`：作为**补充参考对象**

### 6.2 原因总结

#### QGroundControl 更适合作为主要参考对象
- 架构更现代
- 功能组织更清晰
- 更适合作为“Web 二次开发的参考蓝本”
- 任务规划、参数配置、状态展示、模式切换这些逻辑更适合抽取到 Web 方案中
- mentor 当前也已经明确表达：目标是尽量复现 `QGroundControl` 的内容

#### MissionPlanner 更适合作为补充参考
- 功能很全
- 是重要的传统地面站参考
- 但更偏 Windows 桌面工具生态
- 历史包袱更重，不太适合作为 Web 重构的直接架构模板

### 6.3 当前可复用的表述

> `QGroundControl` 用来参考现代 GCS 的结构、页面组织和核心工作流；`MissionPlanner` 主要作为功能补充参考，而不是主结构参考。

## 7. 关于无人船与无人机：最新对齐后的结论

这一块之前讨论很多，但 mentor 最新已经给出明确判断，因此这里必须更新。

### 7.1 之前的疑问
之前主要围绕这些问题展开：
- 无人船和无人机是不是地面站逻辑不一样？
- 是否需要专门为了无人船做功能差异化设计？
- 船是否需要额外的 boat-specific 页面、参数或控制流？

### 7.2 mentor 最新确认
mentor 当前给出的明确判断是：

> **无人船和无人机对于地面站来说，基本一样。**

并进一步明确：

> **当前阶段不需要专门为了无人船去修改功能。就按照无人机地面站的主流做法来做即可。**

### 7.3 这意味着什么

这条结论非常关键，意味着：

1. 当前阶段**不要把产品路线收窄成“无人船专用定制 GCS”**
2. 当前阶段应该优先做的是：
   - 通用 MAVLink GCS 能力
   - 通用控制能力
   - 通用任务规划能力
   - 通用配置能力
3. 无人船目前更多被视为：
   - 当前业务场景
   - 但**不需要因此专门改写一整套功能结构**
4. 功能设计上应以：
   - `QGroundControl` 的通用 GCS 体系为主
   - 而不是从一开始就做一套“船特化 UI”

### 7.4 当前新的统一表述

> 当前阶段，无人船不作为导致地面站功能体系发生根本变化的原因处理；功能上以复现成熟无人机地面站的通用 GCS 能力为主，后续若有特殊船型需求，再做定向扩展。

## 11. 关于 PX4 vs ArduPilot：最新结论已更新

这一块是本次和 mentor 对齐后变化最大的地方之一。

### 11.1 之前的中间判断
之前更偏向：
- 第一阶段先按 ArduPilot 语义设计
- 不要一开始双栈兼容

这个判断在“快速收敛范围”的角度上是合理的，但现在已经被 mentor 的最新要求覆盖。

### 11.2 mentor 最新确认
mentor 当前明确表示：

> **无人系统端最好两套都适配：`PX4-Autopilot` 和 `ArduPilot`。**

并补充说明：
- 虽然参数设置不一样
- 但两者关系比较近
- 可以先尝试适配 `PX4-Autopilot`

### 11.3 这意味着什么

当前项目的产品方向不应再理解为：
- “只面向某一套 autopilot 的特定 GCS”

而应理解为：
- **基于 MavDeck 做通用 Web GCS 升级**
- **目标上兼容 PX4 和 ArduPilot 两套主流 autopilot**
- 但在实施顺序上允许：
  - **先从 PX4-Autopilot 开始适配**
  - 再逐步补齐 ArduPilot 差异兼容

### 11.4 当前新的统一表述

> 当前方向不是“PX4 和 ArduPilot 二选一”，而是“目标上兼容两套主流 autopilot，实施顺序上先从 `PX4-Autopilot` 开始，再逐步补齐 `ArduPilot` 适配”。

### 11.5 对后续架构的影响

这个结论意味着后续做 MavDeck 升级时，应尽量把系统设计成：

1. **通用 GCS 层**
   - 地图
   - 遥测
   - 图表
   - 日志
   - mission UI
   - 参数 UI
   - 连接管理
   - 模式与控制总框架

2. **autopilot 适配层**
   - PX4 模式映射
   - ArduPilot 模式映射
   - PX4 参数组织/元数据处理
   - ArduPilot 参数组织/元数据处理
   - 特定命令差异处理
   - mission / capability 差异处理

也就是说：

> 前端产品形态尽量统一，但底层应逐步形成 PX4 / ArduPilot 的差异适配能力。

## 12. 关于硬件与板卡的认识

已明确：
- `CUAV Pixhawk V6X` 不是“PX4 自家硬件”
- 它是 Pixhawk 生态中的硬件板
- 可以搭配 PX4，也可能搭配 ArduPilot

这意味着：
- 板卡支持某种平台，并不等于地面站可以不区分 autopilot
- 后续仍然要以 autopilot 能力差异作为适配主线

不过在 mentor 已明确“两套都适配”的前提下，板卡问题在当前讨论中的优先级已经下降。

## 13. MavDeck 当前能力与目标之间的差距

### 13.1 当前已有能力
MavDeck 当前已经具备：
- Web 原生基础设施
- MAVLink 基础解析与显示
- 连接能力
- 地图
- 图表
- 遥测展示
- 参数相关基础能力
- 日志记录 / 回放

### 13.2 对标 QGroundControl 仍缺的核心能力
如果目标是“尽量复现 QGroundControl 的核心内容”，那么当前 MavDeck 相比之下仍缺：

1. **更完整的操作闭环**
   - arm / disarm
   - 模式切换
   - 常用 vehicle action

2. **更完整的状态总览**
   - 飞行/航行模式
   - GPS / battery / link / EKF / sensor health
   - 告警和状态提示

3. **更成熟的 setup / 参数配置流程**
   - 不只是参数列表
   - 还包括按业务分组、按步骤配置的能力

4. **更完整的任务规划系统**
   - waypoint 编辑
   - mission 上传 / 下载
   - mission 状态反馈

5. **更完整的页面组织与工作流**
   - Fly / Plan / Setup / Analyze 这类成熟 GCS 工作区概念
   - 更清晰的任务流程组织

6. **多 autopilot 兼容能力**
   - 至少要开始考虑 PX4 与 ArduPilot 的差异适配

### 13.3 当前新的目标表述

> MavDeck 的下一阶段目标，不再只是“补几个功能点”，而是朝着“Web 版 QGroundControl 核心能力复现”这个方向系统推进。

## 15. 分阶段实施思路（当前版本）

这一部分不是最终拍板的里程碑表，但已经基本符合 mentor 最新说法：

> “可以每阶段要复现什么功能的时候和他说一下审核下。”

因此这里给出一个适合后续逐阶段对齐和审查的版本。

### 阶段 1：连接、遥测、基础飞行视图补齐
目标：让 MavDeck 更像一个能用的基础 GCS 主界面

建议优先包含：
- 设备连接稳定化
- 心跳 / 基础状态识别
- 基础遥测总览
- 地图与当前位置/轨迹显示
- 模式、armed 状态、GPS、电池、链路状态显示
- 状态文本 / 告警区
- 基础 action 按钮

阶段产出目标：
- 能像基础版 Fly View 那样工作
- 能支撑基本监控和基础操作

### 阶段 2：控制操作与模式闭环
目标：补齐成熟 GCS 最关键的控制面板能力

建议优先包含：
- arm / disarm
- mode switch
- 常用 action
- guided / auto / rtl 等常用工作流入口
- vehicle 状态变化反馈
- PX4 模式映射先行梳理

阶段产出目标：
- 能完成最核心的地面站控制闭环

### 阶段 3：任务规划（Plan）能力
目标：开始复现 QGC 的 mission 工作流

建议优先包含：
- waypoint 编辑
- mission 上传 / 下载
- mission 列表与地图联动
- 当前任务进度状态显示
- mission 基础校验

阶段产出目标：
- 具备基础 Plan View 能力

### 阶段 4：Setup / 参数配置体系升级
目标：把“参数列表”升级成“可用配置工作流”

建议优先包含：
- 参数读取 / 写入流程稳定化
- 参数搜索、分组、分类
- 常用参数面板
- PX4 / ArduPilot 参数差异适配框架初步建立
- 逐步靠近 Setup 页面思路

阶段产出目标：
- 不只是能改参数，而是有基本的 setup 体验

### 阶段 5：Analyze / 日志 / 回放增强
目标：补齐成熟 GCS 的分析能力

建议优先包含：
- 日志查看增强
- 回放体验增强
- 图表分析增强
- 关键状态历史分析
